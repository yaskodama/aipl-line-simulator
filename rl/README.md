# 強化学習・TinyML

## 1. TinyML —— 手首カメラ画像の認識

実際にレンダリングした手首カメラ画像で、部品を分類する小型NNを学習する。
フレームワークは使わない（純Python / 2355 params）＝実機の Pi でもそのまま動く規模。

```bash
node rl/capture_dataset.mjs      # 手首カメラの実画像から学習データを作る → rl/dataset.json
python3 rl/train_tinyml.py       # 学習 → aipl/tinyml_model.json
```

* 前処理は `src/tinyml.js` に一本化（学習・sim 推論・AIPL 推論が同じ関数を通る）。
* 96x96 RGBA → 中央62%を切出し → 8x8 平均プーリング → 192 次元 → 12(tanh) → 3(softmax)。
* 手首カメラには実機相当の劣化（露出ゆらぎ・WB偏り・センサノイズ）を載せてある。
  劣化が無いと平均色だけで解けてしまい、分類器が何の仕事もしない。

**正直な結果**: 劣化モデル込みでも test accuracy **100%**。3色の彩度が高く色相は
露出変動でも保たれるので、この課題は元々易しい。モデルは本物だが「難しい視覚課題を
解いた」とは読まないこと。

## 2. 強化学習 —— サイクル時間の最小化

Pick&Place を「速く、かつ実機で成立する範囲で」動かす方策を CEM で探索する。

```bash
python3 rl/dofbot_rl.py          # 学習 → rl/rl_policy.json, rl/rl_results.csv
python3 rl/apply_policy.py       # 方策を src/layout.js と aipl/dofbot_xinu.abcl へ書き戻す
node test/cell.mjs               # ← 書き戻したら必ず幾何を検証する
node test/verify.mjs             # ← 実走行（把持誤差・色一致）
node test/cycle_time.mjs         # ← 短縮が実走行でも出ているか実測する
python3 rl/apply_policy.py --restore   # 手調整のベースラインへ戻す
```

**方策 φ (14次元)**: 搬送引込半径 / 搬送高さ / ホバー高さ + 各フェーズ①〜⑪の所要時間。
幾何を変えると全フェーズの関節変位が非線形に変わるので、時間だけを詰める問題ではない。

**報酬** = −(1サイクル時間) − 50×(制約違反)。制約は
サーボ角速度上限 / 可動域 0..180 / 到達性 / 旋回中の棚・既設部品との干渉 /
グリッパの最小開閉時間。**違反が残る解は採用しない**（`apply_policy.py` が拒否する）。

**評価**: 12 スロット全部を巡る決定論的タスク集合（乱数で有利不利が出ない）。

### 結果

| | 手調整 | 学習後 |
|---|---|---|
| サイクル時間 | 6.300 秒 | **5.164 秒 (-18.0%)** |
| 制約違反 | **3.152** | **0.000** |

**重要**: 手調整のベースラインは制約を破っていた。⑥旋回で ID1 を 155° / 1.0 秒 =
ピーク 290°/s 要求しており、これは実機のサーボ速度を超える。学習は⑥旋回を
**遅く**して(1.00→1.63秒)実機で成立させ、他のフェーズを詰めて全体を縮めた。
つまり「速くなった」だけでなく「実機で動くようになった」。

実走行の実測 5.215 秒（設定 5.164 秒）で、短縮はモデル上だけでなく sim でも出ている。

### 前提

`VMAX_DEG_S = 180`（サーボ角速度の上限）は**データシート値ではなく仮定**。
Yahboom の 15kg バスサーボは無負荷で約 300°/s だが、負荷とアーム慣性を見込んで
実用上限として derate した。ここを変えると結論も変わる。
